    Fix 026 & 027: parse error double-emission and continuation prompt

    Date: 2026-05-18 Issues: 026, 027 File: src/plcc/parser/table_cli.py

    Context

    plcc-parse is an orchestrator. It launches plcc-tokens → plcc-trees (which wraps plcc-parser-table), captures their stdout as JSONL, and formats errors as human-readable stderr. Child processes run with --verbose-format=json so the orchestrator can parse their output, even at verbosity level 0.

    plcc-parser-table emits two kinds of output: - Results (stdout JSONL): tree nodes and parse errors — outcomes of the parsing process. - Diagnostics (stderr via verbose.emit_error): tool-internal failures like missing files or non-LL(1) grammars.

    Fix 026 — Remove verbose.emit_error() at parse-result error sites

    Parse errors are results, not diagnostics. They are already emitted on stdout as JSONL records, which parse.py processes into human-readable stderr. The two verbose.emit_error() calls in the parse-error paths of table_cli.py (in the consumed == 0 branch and the except ParseError block) incorrectly treat parse errors as diagnostics, emitting a second JSONL line to stderr.

    Change: Remove those two verbose.emit_error() calls. The two calls for genuine tool failures (cannot load ll1.json, grammar is not LL(1)) are correct and stay.

    Fix 027 — Break on found=="eof" to prevent cascade errors

    When the user enters 23 + interactively (no ^D), the LL(1) parser fails at the eof sentinel: found="eof". table_cli.py currently does cursor += 1 and retries from the next token (OP), which fails with found="OP" — a genuine-looking error. parse.py's only_eof_errors check sees the genuine error and does not return False, so SourceRunner never shows the continuation prompt.

    found="eof" means the parser ran out of input while still expecting more — exactly the condition where continuation is appropriate. There is no meaningful recovery to attempt.

    Change: In the except ParseError block in table_cli.py, when e.found == "eof", emit the error record to stdout and break instead of cursor += 1. This stops the retry loop, leaving only the eof error in the output. parse.py's only_eof_errors check then returns False, and SourceRunner shows the ... continuation prompt. When the user submits with ^D (eof=True), the eof error is treated as real and shown.

    Invariants preserved

    • Genuine first-token errors (e.g., found="OP" when OP is unexpected at the start of <exp>) still advance the cursor and surface errors immediately.
    • Standalone use of plcc-parser-table is unaffected: parse errors still appear on stdout.
    • The only_eof_errors logic in parse.py is unchanged.
