# Cut the v1.0.0 release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the four commits that make the next merge to `main` produce a real `v1.0.0` release through the existing automated pipeline.

**Architecture:** No new code. Four sequential commits on this branch: (1) fix a local/CI tooling version drift, (2) flip a semantic-release config flag, (3) fill in a release-date placeholder, (4) a commit carrying a `BREAKING CHANGE:` marker that both triggers the major version bump and updates a now-stale README claim. Each commit is independently reviewable; together they're one PR.

**Tech Stack:** `pdm`, `python-semantic-release` (PSR), pytest (`bin/test/units.bash`).

## Global Constraints

- The dev dependency on `python-semantic-release` must be constrained to `<10.0.0` — CI's `.github/workflows/release.yml` runs the `@v9` Docker action, which is pinned to core version `9.21.2`; PSR 10 has a different default (`allow_zero_version=False`) that forces bumps CI won't actually perform. Local dry-runs must use the same major version as CI to be trustworthy.
- `major_on_zero` must become `true` in `[tool.semantic_release]` (`pyproject.toml`).
- `docs/whats-new.md`'s release-date placeholder must read `2026-07-06` (today), not `2026-07-XX`.
- The forcing commit must carry a `BREAKING CHANGE:` footer (or `!` after the type) — this is what makes PSR compute a major bump instead of `no_release`.
- After the last commit, push the branch and stop. Do not open or merge a PR — the user handles GitHub PRs and merges by hand.
- `bin/test/units.bash` must stay green after every task (baseline: 1175 passed, 1 skipped).

### Task 1: Constrain the python-semantic-release dev dependency below v10

**Files:**
- Modify: `pyproject.toml:108` (currently `"python-semantic-release>=9.0.0",`)
- Modify: `pdm.lock` (regenerated by `pdm lock`, not hand-edited)

**Interfaces:**
- Consumes: nothing from other tasks.
- Produces: a `pdm`-managed venv running PSR `9.x` locally, matching CI's `@v9` action (core `9.21.2`). Later tasks' `semantic-release version --print` dry-runs depend on this.

- [ ] **Step 1: Edit the dependency constraint**

In `pyproject.toml`, change line 108 from:

```toml
    "python-semantic-release>=9.0.0",
```

to:

```toml
    "python-semantic-release>=9.0.0,<10.0.0",
```

- [ ] **Step 2: Regenerate the lockfile**

Run: `pdm lock`

This rewrites `pdm.lock` to resolve `python-semantic-release` within the new `<10.0.0` ceiling (the CI-matching `9.x` line) instead of `10.5.3`.

- [ ] **Step 3: Sync the local venv**

Run: `pdm install`

- [ ] **Step 4: Verify the resolved version**

Run: `pdm run semantic-release --version`
Expected: a `9.x.x` version string (e.g. `semantic-release, version 9.21.2` or a later 9.x patch) — NOT `10.x.x`.

- [ ] **Step 5: Run the unit test tier**

Run: `bin/test/units.bash`
Expected: `1175 passed, 1 skipped` (same as the pre-existing baseline — this change touches only dev tooling, not `src/`).

- [ ] **Step 6: Commit**

```bash
git add pyproject.toml pdm.lock
git commit -m "$(cat <<'EOF'
chore(deps): constrain python-semantic-release below v10

CI's release workflow pins the python-semantic-release GitHub Action
to @v9 (core 9.21.2), but pyproject.toml's unbounded >=9.0.0 let local
installs drift to 10.5.3 - a version with a different allow_zero_version
default that forces bumps CI's pinned version won't actually perform.
Constrain to <10.0.0 so local dry-runs stay representative of CI.
EOF

```

### Task 2: Flip major_on_zero to allow a v1.0.0 bump

**Files:**
- Modify: `pyproject.toml:121` (currently `major_on_zero = false`, inside `[tool.semantic_release]`)

**Interfaces:**
- Consumes: the `9.x` PSR install from Task 1 (so the dry-run in Step 2 below reflects CI's actual behavior).
- Produces: the config state Task 4's dry-run depends on (`major_on_zero = true` is required for a `BREAKING CHANGE` commit to compute a major rather than minor bump).

- [ ] **Step 1: Edit the flag**

In `pyproject.toml`, change:

```toml
major_on_zero = false
```

to:

```toml
major_on_zero = true
```

- [ ] **Step 2: Dry-run to confirm no retroactive effect**

Run: `pdm run semantic-release version --print`
Expected output ends with:

```
0.67.2
No release will be made, 0.67.2 has already been released!
```

(A `WARNING ... Token value is missing!` line is expected and harmless — no `GH_TOKEN` is set locally.) This confirms flipping the flag alone, with no new releasable commit, does not change or re-release anything already tagged.

- [ ] **Step 3: Commit**

```bash
git add pyproject.toml
git commit -m "$(cat <<'EOF'
chore(release): enable major bumps pre-1.0

major_on_zero=false meant a BREAKING CHANGE commit could only bump
minor pre-1.0. Flipping it lets the next breaking-change commit
produce v1.0.0. Verified this does not retroactively affect already-
released versions: with no new releasable commit, `semantic-release
version --print` still reports 0.67.2 with no release made.
EOF
)"
```
