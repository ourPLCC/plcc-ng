# Issue Conventions

Issues are tracked as files in [issues/](issues/), indexed by [roadmap.md](roadmap.md). Closed issues move to [issues/done/](issues/done/). There is no external tracker; the repository is the source of truth.

## Filing an issue

Always use [bin/issues/new.bash](../bin/issues/new.bash):

```bash
bin/issues/new.bash <slug> [type]
```

It reads the next ID from `issues/.next-id.txt`, creates the file from `issues/TEMPLATE.md`, and increments the counter. Never assign IDs by hand or by scanning the directory.

In the same commit, add an entry to the **Open Issues** section of [roadmap.md](roadmap.md), under the `###` heading matching the issue's type, in this exact two-line format (the scripts below parse it):

```markdown
- **[#NNN](issues/NNN-slug.md) — Title**
  One-line summary of the problem.
```

Commit message: `docs(issues): file NNN - <short title>`.

## The roadmap

[roadmap.md](roadmap.md) is the repository's only forward-looking document. It answers "what is open and what should be picked up next" — never "what shipped." The past already has three owners: `CHANGELOG.md` (generated by semantic-release), GitHub Release notes, and [issues/done/](issues/done/). Do not duplicate it in the roadmap.

Two kinds of sections, with different rules:

- **Open Issues** — an inventory, grouped by conventional-commit type. Its contract is that every entry is open: entries are **removed** on close, and a `###` group with no remaining entries is removed with them.
- **Milestone sections** (e.g. *Path to v1.0*) — ordered task lists tracking progress toward a fixed goal:

  ```markdown
  1. [ ] [#NNN](issues/NNN-slug.md) — one-line rationale for its place in the order.
  ```

  Completed items are **checked off, not removed** (with the link repointed at `issues/done/`), so the list shows progress. The whole section retires when the milestone ships.

## Closing an issue

Close in the same PR that completes the work — as the branch's final commit — so the work and its bookkeeping merge atomically. Use [bin/issues/close.bash](../bin/issues/close.bash):

```bash
bin/issues/close.bash <id>
```

It moves the issue file to `issues/done/`, removes the Open Issues entry (and its group heading if now empty), checks the issue's box in any milestone list, rewrites any `dev-docs/` links to the issue's old path, adjusts the moved file's own links for its new depth, and stages the changes. Review the roadmap before committing — milestone rationale text is prose and is not auto-edited.

Commit message: `docs(issues): close issue N (<short title>), update roadmap`.

**Exception — close on verification.** Some fixes can only be proven by events after the merge (e.g. release-pipeline changes that a real release run must exercise). For those, merge the fix without closing, and close in a follow-up commit once verified. That is a deliberate state, not drift.

## Consistency check

[bin/issues/check.bash](../bin/issues/check.bash) verifies the invariants: every open issue file has an Open Issues entry, every roadmap link resolves and agrees with the file's open/closed state, milestone checkboxes match, and `.next-id.txt` is ahead of every ID ever used. `close.bash` runs it automatically; run it directly after filing an issue or during periodic sweeps. It exits non-zero on any drift.
