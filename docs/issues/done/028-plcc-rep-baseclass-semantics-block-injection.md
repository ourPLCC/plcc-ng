# 028 - plcc-rep: semantics block for alternative rule baseclass should inject into baseclass body

**Type:** feat
**Date:** 2026-05-18

## Description

When `plcc-rep` processes a semantics section, if a block appears for the baseclass of a set of alternative rules, the code in that block should be injected into the generated baseclass body — not ignored or treated as an error.

This allows developers to place abstract method declarations (and other shared code) directly in the baseclass via the familiar `%%%...%%%` syntax.

### Example

Given this grammar:

```
skip WS '\s+'
token NUM '\d+'
token OP '[-*/+]'
%
<exp> ::= <NUM> <exp2>
<exp2>:Exp2More ::= <OP> <exp>
<exp2>:Exp2None ::=
%

Exp
%%%
public void $run() {
    System.out.println("" + eval());
}

public int eval() {
    int x = Integer.parseInt(num.toString());
    return exp2.eval2(x);
}
%%%

Exp2
%%%
public abstract int eval2(int parent);
%%%

Exp2More
%%%
public int eval2(int parent) {
    switch (op.toString().charAt(0)) {
        case '+': return parent + exp.eval();
        case '-': return parent - exp.eval();
        case '*': return parent * exp.eval();
        case '/': return parent / exp.eval();
    }
}
%%%

Exp2None
%%%
public int eval2(int parent) {
    return parent;
}
%%%
```

The `Exp2` rule is a baseclass (the alternatives `Exp2More` and `Exp2None` extend it). The semantics block for `Exp2` declares an abstract method `eval2`. Currently, `plcc-rep` does not inject this code into the generated `Exp2` class, so the abstract declaration is silently lost.

### Expected behavior

The code in `Exp2`'s `%%%...%%%` block is injected into the body of the generated `Exp2` class, enabling polymorphic dispatch through the baseclass:

```java
public abstract class Exp2 extends _Start {
    public abstract int eval2(int parent);
    // ... (plcc-rep boilerplate)
}
```

### Actual behavior

The `Exp2` semantics block is ignored (or an error is raised), and the abstract method is not present in the generated baseclass.

## Notes

- This is the standard mechanism for defining polymorphic interfaces across alternative rule variants.
- The baseclass is generated by `plcc-rep` when a rule has named alternatives (`:AltName` syntax). The baseclass name matches the non-terminal name (e.g., `<exp2>` → `Exp2`).
- Injection should work the same way as for concrete rule classes — the block contents are placed inside the class body.
- If the baseclass block contains abstract methods, the generated class must be declared `abstract`.

### All modifiers must work for baseclasses

The fix should ensure that every modifier supported for concrete classes also works when the target class is a baseclass:

| Modifier | Placement | Languages |
| -------- | --------- | --------- |
| *(none)* | Inside class body | Java, Python |
| `:init` | Inside constructor body | Java, Python |
| `:import` | Import statements at top of file | Java, Python |
| `:top` | Before class declaration | Python |
| `:class` | Class-level scope (outside methods) | Python |
| `:file` | Replaces the entire generated file | Java, Python |

Each of these should behave identically whether the target class is a concrete rule class or a baseclass generated from alternative rules.
