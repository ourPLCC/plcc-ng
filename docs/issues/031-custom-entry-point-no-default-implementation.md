# 031 - Custom entry point with no implementation crashes at runtime

**Type:** fix
**Date:** 2026-05-22

## Description

When a semantics section declares a custom `entry_point` (not the language default `_run` / `$run`) but provides no method body for it, the generated code crashes at runtime. The `_Start` base class generated by issue 015 only provides a default `_run()` / `$run()` method, so it does not satisfy a custom entry point name.

Python example: if a semantics section sets `entry_point: evaluate`, `main.py` calls `tree.evaluate()` but neither `_Start` nor any generated class defines it.

Java example: if a semantics section sets `entry_point: evaluate`, `Main.java` calls `getMethod("evaluate")` via reflection, which raises `NoSuchMethodException` because `_Start.$run()` does not match.

## Steps to Reproduce

1. Write a grammar with a semantics section that sets a custom entry point (`entry_point: evaluate`) but defines no method body for it on any class.
2. Emit code (`plcc-python-emit` or `plcc-java-emit`).
3. Run the emitted `main.py` or compiled `Main.class` — it crashes with `AttributeError` (Python) or `NoSuchMethodException` (Java).

## Notes

- Issue 015 intentionally only addressed the "no semantics section" case, where the default entry point always applies.
- The fix needs design: options include (a) emitting `_Start` with the resolved `entry_point` name instead of the hardcoded default, (b) emitting an alias method alongside the default, or (c) something else. Needs a spec before implementation.
- The custom `entry_point` feature lives in the semantics section parsing and is threaded through the model into the emitters via `model['semantic_sections'][0]['entry_point']`.
