    v1.0 Readiness Criteria Implementation Plan

    For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

    Goal: Publish a durable dev-docs/v1.0-criteria.md recording the agreed v1.0 criteria, the audit evidence for each, and the project owner's stability declarations — and point issue #112 at it.

    Architecture: Two docs-only changes: (1) a new dev-docs page plus its mkdocs-dev.yml nav entry, (2) a short pointer added to issue #112's Notes section. No code changes; no pytest/bats involved. Verification is bin/issues/check.bash (issue/roadmap consistency) and an mkdocs build --strict against mkdocs-dev.yml (catches broken nav/links).

    Tech Stack: Markdown, MkDocs (Material theme), existing bin/issues/*.bash tooling.

    Global Constraints

    • Criteria content must match docs/superpowers/specs/2026-07-06-112-v1-criteria-design.md exactly — same 7 criteria, same "Met"/"Met (declared)" statuses, same evidence.
    • Do not close issue #112 and do not touch dev-docs/roadmap.md — the release cutover remains open work under #112, so its roadmap line stays as-is.
    • Do not run bin/issues/close.bash in this plan.
    • Commits use conventional-commit style (docs(...): ...); no [skip ci] needed since these paths are already CI-exempt.

    • [ ] Step 1: Write dev-docs/v1.0-criteria.md

    Create the file with this exact content:

    # v1.0 Readiness Criteria

Declared 2026-07-06, tracked in [issue #112](issues/112-first-major-release.md).

## Criteria

1. **No known correctness bugs in the core pipeline (scan → parse → emit)** — MET
   No open bug-type issues; recent `fix:` commits are release-tooling only, not pipeline correctness.
2. **Stable CLI surface: `plcc-scan`, `plcc-parse`, `plcc-rep`, `plcc-diagram`** — MET (declared)
   Command names, flags, and output formats are committed to; changes require a major version bump.
3. **Stable spec syntax and semantics** — MET (declared)
   No planned breaking changes to the `.plcc` file format.
4. **Stable language extensions: Python, Java, JavaScript, Haskell** — MET (declared)
   All four are v1-supported; none are experimental.
5. **End-user docs cover the full workflow** — MET
   Install, quickstart, language guide, and CLI reference are all present under `docs/`.
6. **Release SOP complete and tested** — MET
   [`dev-docs/release-sop.md`](release-sop.md) written and exercised (issue #130).
7. **CI green with Python + Java e2e happy-path coverage** — MET
   `tests/bats/e2e/happy-path.bats` (Python), `tests/bats/e2e/languages-java.bats` (Java); CI green on `main`.

All seven criteria are met as of this writing.

## Remaining before v1.0.0 ships

These stay open under issue #112 — out of scope for this document:

- Flip `major_on_zero` in `pyproject.toml` (or otherwise force a `1.0.0` tag)

- Update `docs/whats-new.md`'s `2026-07-XX` placeholder to the actual release date
- Run `bin/release/verify.bash` against the new tag
