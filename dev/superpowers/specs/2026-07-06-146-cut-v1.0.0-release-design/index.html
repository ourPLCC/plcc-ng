    Design: Cut the v1.0.0 release

    Issue: dev-docs/issues/146-cut-v1.0.0-release.md Date: 2026-07-06

    Problem

    All seven v1.0 readiness criteria (issue #112, dev-docs/v1.0-criteria.md) are met. What remains is the actual cutover: get a real v1.0.0 tag published through the existing automated release pipeline (dev-docs/release-sop.md).

    Releases here are fully automated: merging releasable conventional commits to main is the release. python-semantic-release (PSR) decides the version from commit types since the last tag, with major_on_zero = false in pyproject.toml — meaning even a BREAKING CHANGE: commit only bumps minor pre-1.0. Nothing forces a v1.0.0 tag on its own, and every commit since the last tag (v0.67.2) is docs/chore, so a plain merge right now would release nothing at all.

    Investigation finding: local/CI PSR version drift

    pyproject.toml pins the dev dependency as python-semantic-release>=9.0.0 (unbounded). Locally this resolved to 10.5.3, while CI's .github/workflows/release.yml uses the python-semantic-release/python-semantic-release@v9 Docker action, which is pinned to core version 9.21.2 — confirmed via the action's pyproject.toml at the v9 tag.

    This matters because PSR 10 introduced allow_zero_version (default False), which forces a version bump out of 0.x on any commit past a tag, regardless of commit type — different behavior from 9.21.2, which has no such mechanism and correctly stays quiet when nothing is releasable (confirmed by reading 9.21.2's algorithm.py, which has no allow_zero_version reference). A local dry-run using the unpinned resolved version (10.5.3) is therefore not representative of what CI will actually do. This is fixed as part of this work (see Plan, step 1) so local tooling and CI stay in sync going forward.

    Validation performed

    Using an isolated scratch clone (not this branch) and a venv with python-semantic-release==9.21.2 installed (matching CI exactly):

    1. Baseline (main as-is, major_on_zero = false): semantic-release version --print → 0.67.2, "No release will be made, 0.67.2 has already been released!" — matches the SOP's documented behavior.
    2. Flag flipped alone (major_on_zero = true, no new commit): same result — 0.67.2, no release. Confirms flipping the flag does not retroactively reinterpret or re-release past history; it only changes how future commits are evaluated.
    3. Flag flipped + a trigger commit (feat: ... with a BREAKING CHANGE: footer): semantic-release version --print → 1.0.0.
    4. Changelog generation: diffing CHANGELOG.md before/after showed a purely additive ## Unreleased section inserted after the header — no existing entries altered.

    This directly answers the issue's own caution ("confirm this doesn't retroactively reinterpret past commits when semantic-release next runs") with evidence rather than inference.

    Plan

    Four commits on this branch, landing together in one PR merged to main (merging is the release — no separate manual release step):

    1. chore(deps): constrain python-semantic-release below v10 Change pyproject.toml's dev dependency to python-semantic-release>=9.0.0,<10.0.0, run pdm lock/pdm install to update pdm.lock, confirm bin/test/units.bash still passes.

    2. chore(release): enable major bumps pre-1.0 Flip major_on_zero = false → true in [tool.semantic_release] in pyproject.toml.

    3. docs: set v1.0.0 release date in What's New In docs/whats-new.md, replace the first entry's 2026-07-XX with 2026-07-06. The heading (PLCC-ng v1.0.0) and the <!-- last-covered: v1.0.0 --> marker already match; only the date placeholder changes.

    4. feat!: declare plcc-ng v1.0.0 (with a BREAKING CHANGE: footer) The commit that actually triggers the major bump. Real content: README.md currently states plcc-ng is "experimental — no compatibility guarantees with plcc until a stable 1.0 release." That claim becomes false at this moment, so this commit rewrites it to declare stability, referencing dev-docs/v1.0-criteria.md. This mirrors the repo's own precedent (1ce9cd58, feat: trigger release / BREAKING CHANGE: ...), previously used the same way to coerce a major release.

    Push the branch and stop — per established workflow, PR creation and merge are handled by the user, not automated here.

    After merge (not part of this branch)

    • Watch Actions → Release for the run.
    • Once the GitHub Release for v1.0.0 is published, run bin/release/verify.bash v1.0.0 (without --no-install) per the SOP, to confirm PyPI, the GitHub Release, and versioned docs all landed and the package installs and smoke-tests cleanly from real PyPI.
    • If any stage fails, recovery is per the table in dev-docs/release-sop.md (e.g. "Re-run failed jobs" for a failed semantic-release step before tagging; the republish dispatch for anything after tagging).

    Testing

    • bin/test/units.bash after step 1 (dependency constraint change) to confirm nothing in the local toolchain broke.
    • The PSR dry-run validation above already exercises the version-computation logic end-to-end against the real CI version; no additional automated test is meaningful for a one-time release cutover beyond the existing bin/release/verify.bash post-merge check.

    Out of scope

    • Any further changes to release-pipeline mechanics beyond the version-drift fix above.
    • Anything not already covered by the seven v1.0 criteria (issue #112) — this issue is the cutover only.
